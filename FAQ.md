# FAQ — xWRT

## Общие вопросы

### Нужно ли настраивать каждое устройство отдельно?

Нет. xWRT работает на уровне роутера — весь трафик в сети проходит через прокси автоматически. Браузеры, приложения, смарт-ТВ, консоли — всё без дополнительных настроек.

---

### Чем xWRT отличается от XKeen?

XKeen разработан для Keenetic с Entware и использует iptables + tun-интерфейс. xWRT написан для OpenWRT и использует **TProxy + nftables** — более современный подход.

Ключевое отличие: XKeen требует явно перечислять порты для каждого приложения (Telegram: 443, 5222; WhatsApp: 443, 3478 и т.д.), потому что iptables REDIRECT работает только с TCP и требует явного указания UDP-портов. TProxy захватывает весь TCP и UDP сразу — никакие порты перечислять не нужно.

---

### Все ли порты проксируются?

Да, всё TCP и UDP, кроме:

- **UDP/TCP 53 (DNS)** — явно исключён, роутерный резолвер работает как обычно
- **Приватные диапазоны** — `10.0.0.0/8`, `192.168.0.0/16`, `172.16.0.0/12` и другие всегда идут напрямую
- **UDP 443 (QUIC)** — заблокирован (подробнее ниже)
- **Клиенты из `exclude.lst`** — их трафик полностью обходит прокси

---

### Будут ли работать звонки в Telegram и WhatsApp?

В большинстве случаев — да. Звонки работают через UDP, а TProxy захватывает весь UDP без ограничений по портам.

Единственное условие: **прокси-сервер должен поддерживать UDP-релей** (VLESS+xudp, Shadowsocks, Trojan и т.п.). В шаблонах xWRT принудительно включён `udp: true` для всех серверов из подписки — если сервер поддерживает UDP, звонки пройдут.

Если звонки не работают — проблема на стороне прокси-провайдера, а не роутера.

---

### Почему заблокирован QUIC (UDP port 443)?

QUIC — протокол Google поверх UDP/443, используется в HTTP/3. Браузеры предпочитают QUIC, но большинство прокси-протоколов не умеют его ретранслировать прозрачно, из-за чего сайты могут открываться медленнее или не открываться вообще.

Блокировка QUIC (`REJECT-DROP`) заставляет браузеры откатиться на TCP/TLS, который корректно проксируется. На пользовательском опыте это незаметно.

Telegram и WhatsApp **не используют UDP/443** для голосовых звонков — они работают на других портах, поэтому блокировка QUIC на звонки не влияет.

## Режимы работы

### В чём разница между режимами vpn и direct?

| | `vpn` | `direct` |
|---|---|---|
| Незаблокированные сайты | Через прокси | Напрямую |
| Заблокированные сайты (RKN) | Через прокси | Через прокси |
| Российские ресурсы (category-ru) | Напрямую | Напрямую |
| Финальное правило (MATCH) | PROXY | DIRECT |

**vpn** — проще и надёжнее, все сайты открываются через прокси. Минус: чуть выше нагрузка на VPN-сервер и пинг на незаблокированных ресурсах.

**direct** — трафик к незаблокированным сайтам идёт напрямую, скорость выше. Минус: нужно следить за актуальностью списков блокировок.

---

### Режим direct не разблокирует нужный сайт

Сайт может отсутствовать в списках блокировок, которые используют шаблоны. Добавьте его вручную в секцию `user@classical` в шаблоне:

```yaml
user@classical:
  type: inline
  behavior: classical
  payload:
    - DOMAIN-SUFFIX,example.com
```

Файл: `/etc/mihomo/templates/config-direct.yaml`  
После правки: `xwrt mode direct`

## Диагностика

### Как проверить что прокси работает?

```sh
xwrt status         # статус процесса, nftables, маршрутизация
xwrt log 50         # последние 50 строк лога
xwrt debug          # включить подробный лог info
xwrt watch          # смотреть лог в реальном времени
```

Также откройте веб-интерфейс `http://<IP роутера>:9090/ui` — там видны активные соединения и статус прокси-серверов.

---

### Сайт не открывается даже через прокси

1. Убедитесь что выбран рабочий прокси-сервер в веб-интерфейсе (`:9090/ui`)
2. Проверьте health-check — сервер должен отвечать на тест
3. Включите отладку: `xwrt debug`, затем `xwrt watch` и попробуйте открыть сайт — увидите через какой proxy-group идёт трафик
4. Обновите подписку: `xwrt update-sub`

---

### mihomo не запускается

```sh
xwrt log 50   # смотрим ошибки
```

Частые причины:
- Неверный формат `config.yaml` — пересоздайте из шаблона: `xwrt mode vpn`
- Порт 7892 или 7893 занят другим процессом
- Бинарник повреждён — `xwrt update-bin`

---

### Подписка не загружается / proxy-sub пустой

```sh
xwrt update-sub   # принудительная перезагрузка через API
```

Если не помогает:
```sh
xwrt sub          # проверить сохранённый URL
xwrt sub <url>    # обновить URL и пересоздать конфиг
```

---

### Высокий пинг или медленная скорость

- В режиме `vpn` — нормально для зарубежных серверов. Переключитесь на `direct` если нужна скорость к российским ресурсам
- Проверьте latency серверов в веб-интерфейсе `:9090/ui` — выберите ближайший
- Включите `unified-delay: true` (уже включено в шаблонах) — показывает реальный RTT

## Совместимость

### Работает ли xWRT вместе с zapret?

Да, без конфликтов. mihomo работает в цепочке `PREROUTING` (приоритет mangle), zapret — в `FORWARD/OUTPUT` (приоритет filter). Порядок обработки:

```
клиент → PREROUTING (mihomo) → [VPN туннель]
                             ↘ FORWARD → zapret DPI-обход → интернет
```

Если VPN-сервер использует порт 443 и zapret перехватывает весь TCP/443 в OUTPUT — добавьте IP VPN-серверов в исключения zapret.

---

### Можно ли исключить отдельные устройства из прокси?

Да. Добавьте IP устройства в `/etc/mihomo/exclude.lst`:

```
# Телевизор — напрямую
192.168.1.100

# Игровая консоль
192.168.1.101
```

После изменения: `xwrt restart`

## Обслуживание

### Как обновить mihomo?

```sh
xwrt update-bin
```

Скрипт проверит текущую и последнюю версии, скачает и заменит бинарник, перезапустит сервис.

---

### Как часто обновляются правила и подписка?

По умолчанию в шаблонах:
- **Подписка** (`proxy-providers`) — каждые 60 минут
- **Rule-sets** (`rule-providers`) — каждые 24 часа

Принудительное обновление без перезапуска:
```sh
xwrt update-sub    # перезагрузить подписку через API
xwrt update-rules  # обновить все rule-sets через API
```

---

### Нагружает ли xWRT флеш-память роутера?

Минимально. Логи пишутся в RAM (`logread`). На флеш записывается только:
- Обновление кешированной подписки (`proxy-sub.yml`) — ~раз в час
- Обновление rule-sets — ~раз в сутки

`log-level: warning` по умолчанию — на флеш ничего не пишется.
